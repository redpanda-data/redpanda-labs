== Overview

This lab demonstrates how to build a streaming data lakehouse using Redpanda's Wasm data transforms and Iceberg integration. You'll deploy a transform that performs true 1:N fanout, parsing batch messages and routing individual updates from a single input topic to multiple domain-specific output topics, each configured as an Iceberg table for analytics.

A Go producer sends complete JSON batches to an input topic. The transform dynamically registers JSON schemas in Schema Registry during initialization, then parses each batch, extracts individual updates, and fans them out to the appropriate Iceberg-enabled topics. Redpanda automatically validates messages against the registered schemas and writes to Iceberg tables.

This approach offers several advantages:

* No external ETL. Routing happens inside Redpanda brokers using Wasm transforms.
* Dynamic schema registration. The transform registers schemas automatically during initialization, eliminating manual setup steps.
* Schema Registry validates every routed message using JSON Schema.
* Iceberg tables are immediately queryable by Spark.
* Operational simplicity. A single Redpanda cluster handles routing, validation, and storage.

Consider this approach for use cases like:

* Multi-tenant data routing to separate lakehouse tables
* CDC events routed by table name to analytics destinations
* Event-driven architectures with domain separation
* Real-time data warehouse ingestion

== Run the lab

=== Set up the environment

. Clone the repository:
+
[,bash]
----
git clone https://github.com/redpanda-data/redpanda-labs.git
----

. Change into the lab directory:
+
[,bash]
----
cd redpanda-labs/data-transforms/go/iceberg-fanout
----

. Set environment variables for Redpanda and Console versions:
+
[,bash,subs="attributes+"]
----
export REDPANDA_VERSION=v{latest-redpanda-version}
export REDPANDA_CONSOLE_VERSION=v{latest-console-version}
----

. Start all services with Docker Compose:
+
[,bash]
----
docker compose up -d
----
+
This starts Redpanda, MinIO (S3-compatible storage), Iceberg REST catalog, Redpanda Console, and Spark with Jupyter.

. Create an rpk profile for the lab:
+
[,bash]
----
rpk profile create iceberg-fanout --from-profile profile.yml
----

=== Create topics

. Create the input topic for batched events (no Iceberg configuration):
+
[,bash]
----
rpk topic create events
----

. Create Iceberg-enabled output topics with Schema Registry integration:
+
[,bash]
----
rpk topic create orders --topic-config=redpanda.iceberg.mode=value_schema_latest
rpk topic create inventory --topic-config=redpanda.iceberg.mode=value_schema_latest
rpk topic create customers --topic-config=redpanda.iceberg.mode=value_schema_latest
----
+
The `value_schema_latest` mode tells Redpanda to automatically validate incoming messages against the latest JSON Schema from Schema Registry and write to Iceberg tables.

. Verify all topics are created:
+
[,bash]
----
rpk topic list
----

=== Build and deploy the transform

The transform automatically registers JSON schemas in Schema Registry during initialization.

NOTE: The transform source code is provided in `transform/transform.go`. It demonstrates how to use the Schema Registry client to register schemas dynamically. If you were creating a new transform from scratch, you would start with `rpk transform init` to scaffold the project.

. Build and deploy the transform:
+
[,bash]
----
cd transform
rpk transform build
rpk transform deploy -i events -o orders -o inventory -o customers
----
+
During deployment, the transform registers three JSON schemas with Schema Registry:
+
* `orders-value` - validates order records
* `inventory-value` - validates inventory records
* `customers-value` - validates customer records
+
The `CreateSchema` method is idempotent, so redeploying the transform is safe and won't create duplicate schemas.

. Verify the transform is deployed:
+
[,bash]
----
rpk transform list
----
+
Example output:
+
[,role="no-copy"]
----
NAME           INPUT TOPIC  OUTPUT TOPIC                  RUNNING  LAG
events-fanout  events       orders, inventory, customers  1 / 1    0
----

. Check the transform logs to confirm schema registration:
+
[,bash]
----
rpk transform logs events-fanout
----
+
You should see log entries like:
+
[,role="no-copy"]
----
Registering schemas in Schema Registry...
Registered schema for orders: ID=1
Registered schema for inventory: ID=2
Registered schema for customers: ID=3
Starting multi-topic fanout transform
----

. Verify schemas are registered in Schema Registry:
+
[,bash]
----
rpk registry schema list
----
+
Example output:
+
[,role="no-copy"]
----
SUBJECT          VERSION  ID    TYPE
customers-value  1        3     JSON
inventory-value  1        2     JSON
orders-value     1        1     JSON
----
+
TIP: The Iceberg-enabled output topics use `value_schema_latest` mode, which tells Redpanda to automatically validate incoming messages against the latest JSON Schema from Schema Registry before writing to Iceberg tables.

=== Build and run the producer

The lab includes a Go producer (using franz-go) that reads JSON batches and sends complete batch messages to the events topic for the transform to parse.

. Build the producer:
+
[,bash]
----
cd ..
cd producer
go mod download
go build -o produce-events
----

. Understand the sample data format:
+
The `sample-batches.txt` file contains JSON batches. Each batch has multiple table updates:
+
[,json]
----
{
  "updates": [
    {"table": "orders", "data": {"order_id": "ORD-1001", "amount": 99.99, "timestamp": 1708185600000}},
    {"table": "inventory", "data": {"product_id": "P100", "quantity": 50, "timestamp": 1708185600000}},
    {"table": "customers", "data": {"customer_id": "C5001", "name": "Alice Johnson", "timestamp": 1708185600000}}
  ]
}
----
+
The producer sends the entire batch JSON as a single message to the events topic. The transform then parses each batch and fans out the individual updates to their target topics.

=== Produce sample data

[,bash]
----
cat ../sample-batches.txt | ./produce-events
----

This sends all 10 batches from the file. The transform will fan out these 10 batch messages into 30 individual messages (10 to each output topic).

=== Verify routing in Redpanda Console

. Open Redpanda Console at http://localhost:8080.

. Navigate to Topics and check each topic to see the routed messages and verify the transform is working correctly:
+
* `events`: Input topic containing 10 complete batch JSON messages (each with "updates" array)
* `orders`: Output topic containing 10 individual order updates
* `inventory`: Output topic containing 10 individual inventory updates
* `customers`: Output topic containing 10 individual customer updates
+
Notice the fanout: 10 batch messages became 30 individual messages across the three output topics.

. Check transform logs to verify the fanout processing:
+
[,bash]
----
rpk transform logs events-fanout
----
+
You should see log entries showing:
+
* "Processing batch with 3 updates" (for each input batch)
* "Routed update to orders", "Routed update to inventory", "Routed update to customers"
* "Successfully fanned out 3 updates" (for each batch)

=== Query Iceberg tables with Spark

The Iceberg-enabled topics automatically write data to object storage, with the Iceberg REST catalog tracking table metadata (schemas, snapshots, manifest files). When you query these tables in Spark, it uses the REST catalog to discover table structures and locate the data files in MinIO.

. Open Jupyter at http://localhost:8888.

. Open the provided notebook `Query-Fanout-Tables.ipynb` which contains pre-built queries for analyzing the routed data, or create your own notebook with the following queries:

==== Count records in each table

[,python]
----
# Initialize Spark session (if not already done)
from pyspark.sql import SparkSession

# Count records in each Iceberg table
print("Orders:", spark.sql("SELECT COUNT(*) FROM lab.redpanda.orders").first()[0])
print("Inventory:", spark.sql("SELECT COUNT(*) FROM lab.redpanda.inventory").first()[0])
print("Customers:", spark.sql("SELECT COUNT(*) FROM lab.redpanda.customers").first()[0])
----

==== View routed data

[,python]
----
# Show recent orders
spark.sql("""
    SELECT order_id, amount, timestamp
    FROM lab.redpanda.orders
    ORDER BY timestamp DESC
    LIMIT 5
""").show()

# Show inventory levels
spark.sql("""
    SELECT product_id, quantity, timestamp
    FROM lab.redpanda.inventory
    ORDER BY timestamp DESC
    LIMIT 5
""").show()

# Show customers
spark.sql("""
    SELECT customer_id, name, timestamp
    FROM lab.redpanda.customers
    ORDER BY timestamp DESC
    LIMIT 5
""").show()
----

==== Verify schema enforcement

[,python]
----
# Show table schemas to verify correct data types
print("Orders schema:")
spark.sql("DESCRIBE lab.redpanda.orders").show()

print("\nInventory schema:")
spark.sql("DESCRIBE lab.redpanda.inventory").show()

print("\nCustomers schema:")
spark.sql("DESCRIBE lab.redpanda.customers").show()
----

NOTE: It may take a few seconds for data to appear in Iceberg tables after producing. Redpanda writes to Iceberg based on the `iceberg_target_lag_ms` setting (5 seconds in this lab).
